ADR-0005: Physical-Tenant Routing of Authorization Reads in the Service and CSL Adapter Layers

Status​

Proposed

Deciders​

Proposed for review by the Physical Tenants Identity squad (the ADR-0003 deciders):

  • Sebastian Bathke (@megglos) — proposer
  • Ben Sheppard (@Ben-Sheppard)
  • Tobias Metzke (@tmetzke)
  • Deepthi Akkoorath (@deepthidevaki)
  • Patrick Wunderlich (@p-wunderlich)

Context​

ADR-0003 established that the physical-tenant (PT) id is stamped on the request before Spring Security's filter chain runs and is available throughout the request via PhysicalTenantContext.current() and the @PhysicalTenantId argument resolver. Non-prefixed cluster paths (/v2/...) fall back to default.

On top of that, a per-PT read infrastructure already exists on main:

  • ES/OS — PhysicalTenantSearchClientReadersConfiguration builds a Map<String, SearchClientReaders> (one reader bundle per configured PT, each bound to that PT's secondary-storage connection).
  • RDBMS — the equivalent per-PT Map<String, RdbmsTenantReaders> introduced in #54474.
  • ServiceRegistry exposes per-PT services (authorizationServices(physicalTenantId), userServices(physicalTenantId), …); CamundaSearchClients routes data reads to the in-context PT via withPhysicalTenant(id), which selects tenantReaders.get(id).

As a result the authorization query API is already PT-correct end to end: AuthorizationController.searchAuthorizations(@PhysicalTenantId …) → ServiceRegistry.authorizationServices(pt) → CamundaSearchClients (PT-scoped readers).

The gap is the authorization decision read — the lookup that the Camunda Security Library (CSL) performs to decide whether a principal may access a resource. That read is served by a singleton AuthorizationReader bean hardwired to the default PT:

  • ES/OS — SearchClientReaderConfiguration.authorizationReader() resolves the reader via requireDefault(...), i.e. the default entry of the per-PT reader map only.
  • RDBMS — RdbmsConfiguration builds the equivalent default-only AuthorizationReader.

Two call paths consume that singleton (#55252):

  1. Control-plane (Spring Security permission checks): hasPermission() → CSL ResourcePermissionService → AuthorizationRepositoryAdapter.findAuthorizations() → AuthorizationReader.search().
  2. Data-plane (result authorization inside CamundaSearchClients): doSearch/doGet → ResourceAccessDelegatingController → AuthorizationChecker → AuthorizationScopeRepositoryPort (SearchAuthorizationScopeRepository) → AuthorizationReader.

So even though a search for PT B's data reads from PT B's storage, the access check for that search consults PT default's authorization records. In a multi-PT deployment, authorization for every non-default PT is evaluated against the wrong tenant's data — on both paths, for both backends.

This decision is constrained by the Slice 1 (#54728) principle that CSL never learns about physical tenants: CSL exposes generic extension points (ports), and all PT resolution lives on the OC side in the host-supplied adapters and bean wiring. The basic-auth path already follows this shape — BasicAuthUserDetailsAdapter (an OC-supplied BasicAuthUserDetailsPort) resolves PhysicalTenantContext.current() and selects serviceRegistry.userServices(physicalTenantId) at call time. The authorization adapters are the missing counterpart.

Both consumers currently inject the same singleton authorizationReader bean, but they run in different execution contexts, and that difference is decisive:

  • The control-plane check runs only inside a Spring Security filter, on the request thread, where PhysicalTenantContext.current() is populated by the pre-security filter (ADR-0003).
  • The data-plane check runs inside CamundaSearchClients, which is invoked both from the request path and from the engine. Batch operations materialise their item set by querying secondary storage from the broker — ProcessInstanceItemProvider / IncidentItemProvider / DecisionInstanceItemProvider call searchClientsProxy.withSecurityContext(creatorAuth).search…(…), which runs the resource-access check off the request thread, where no PhysicalTenantContext is bound and PhysicalTenantContext.current() falls back to default.

The engine is already structured per physical tenant (one PartitionManager per PT; partition group = PT id), and #55401 scopes the engine's data reads to the partition's PT via searchClientsProxy.withPhysicalTenant(partitionGroup). But withPhysicalTenant today swaps only the data readers, not the ResourceAccessController — so the authorization read on that path still resolves through a process-wide controller and a default-pinned reader. A pure thread-local resolution therefore cannot be correct on the data-plane: there is no request context in the engine to read. The data-plane must resolve the PT instance-bound — from the PT the executing CamundaSearchClients is scoped to — not from a thread-local.

(An earlier spike, #55336, concluded a single thread-local mechanism sufficed, having examined only the request path; the engine batch-operation path above is the off-request-thread consumer that disproves that conclusion. It also matches the isolation principle agreed by the squad: a per-PT engine incarnation must read only its own PT's secondary storage.)

Decision​

Resolve authorization reads to the in-context physical tenant with two mechanisms, chosen by where each consumer executes, both built on a single shared building block, with CSL kept physical-tenant-agnostic. The default-pinned singleton authorizationReader is removed.

Shared building block — the per-PT AuthorizationReader. Each configured PT already owns an AuthorizationReader inside its reader bundle (SearchClientReaders.authorizationReader() / RdbmsTenantReaders.authorizationReader()). Both mechanisms select one of these; neither falls back to default implicitly.

  1. Data-plane → instance-bound, via CamundaSearchClients. CamundaSearchClients holds a Map<String, ResourceAccessController> and withPhysicalTenant(pt) selects both the PT's data readers and the PT's ResourceAccessController (today it switches only the readers). Each per-PT controller is the existing chain — DocumentBased / RdbmsResourceAccessController → DefaultResourceAccessProvider → CSL AuthorizationChecker → SearchAuthorizationScopeRepository — constructed over that PT's AuthorizationReader. Because the controller is bound to the search client's PT, the authorization read follows the data read in every caller: the request path (search client scoped from PhysicalTenantContext) and the engine (search client scoped from the partition's PT, #55401). No thread-local is consulted here.

  2. Control-plane → request-scoped, in the adapter. Spring Security permission checks run only on the request thread (the engine never calls hasPermission). The AuthorizationRepositoryAdapter (OC's AuthorizationRepositoryPort impl) resolves PhysicalTenantContext.current() and reads from that PT's AuthorizationReader. This adapter reads with ResourceAccessChecks.disabled() — it is the permission-computation data source, so it does not (and must not) go through a ResourceAccessController; it routes at the AuthorizationReader layer, the same building block the data-plane controller sits on.

CSL is untouched — all PT resolution lives in OC's port adapters and bean wiring. Net contract: for any caller carrying PT X (the request's PT, or a partition's PT in the engine), both the permission check and the result-authorization check read PT X's authorization records.

Consequences​

Positive​

  • Authorization is enforced against the correct PT on both planes, both backends, and in both execution contexts (request and engine) — closing #55252 including the batch-operation path.
  • Upholds the engine isolation principle: a per-PT engine incarnation reads only its own PT's authorization data; no usual execution path reaches another PT's secondary storage.
  • No implicit default routing on the data-plane — the PT is always explicit (the search client's scope), in line with the agreed direction to set the PT explicitly and never silently fall back to default.
  • CSL stays physical-tenant-agnostic (Slice 1); the per-PT AuthorizationReader reuses the per-PT reader infrastructure already on main (ES/OS, and RDBMS via #54474).

Negative / trade-offs​

  • Two mechanisms rather than one — justified by genuinely different execution contexts (request-only control-plane vs request-and-engine data-plane), but more to understand than a single bean swap.
  • The data-plane change requires per-PT instances of the resource-access chain (ResourceAccessController → ResourceAccessProvider → AuthorizationChecker → AuthorizationScopeRepository), constructed alongside the per-PT readers and held in CamundaSearchClients. The components are thin wrappers, so the cost is wiring, not logic.
  • The control-plane still resolves via the thread-bound PhysicalTenantContext. Acceptable because that path is request-only; a hypothetical non-request permission check would need the PT supplied explicitly.

Neutral​

  • No change to the authorization query API (already PT-routed) and none to CSL.
  • Single/default-PT deployments are unchanged (everything resolves to the one default PT).

Alternatives Considered​

  1. Single thread-local routing AuthorizationReader for both planes — re-point the one authorizationReader bean to a reader that resolves PhysicalTenantContext.current() and both ports inherit it. Superseded. A spike (#55336) adopted this after concluding the data-plane read always runs on the request thread — but that examined only the request path. Batch operations run authorization-checked secondary-storage searches in the engine, off the request thread, where PhysicalTenantContext.current() falls back to default. Thread-local is correct for the request-only control-plane but unsafe for the data-plane, which is why the data-plane is resolved instance-bound instead. (It also implicitly routes to default, which the squad agreed to avoid.)

  2. Embed the routing inline in the host-side adapter(s) (à la BasicAuthUserDetailsAdapter). This is in fact how the control-plane is resolved in the decision above — the adapter selects the per-PT reader from PhysicalTenantContext. It does not help the data-plane, whose check runs in the engine with no request context; hence the separate instance-bound mechanism there.

  3. Carry the PT inside CSL's SecurityContext so it could be threaded explicitly through CSL. Rejected: it makes CSL's context object physical-tenant-aware, violating the Slice 1 principle that CSL stays PT-agnostic.

  4. Do nothing / defer past alpha3. Rejected: multi-PT authorization is incorrect without this, and it is also an engine isolation concern. Final alpha3 scoping is confirmed on #55252 / #54728.
