ADR-0005: Physical-Tenant Routing of Authorization Reads in the Service and CSL Adapter Layers

Status​

Proposed

Deciders​

Proposed for review by the Physical Tenants Identity squad (the ADR-0003 deciders):

  • Sebastian Bathke (@megglos) — proposer
  • Ben Sheppard (@Ben-Sheppard)
  • Tobias Metzke (@tmetzke)
  • Deepthi Akkoorath (@deepthidevaki)
  • Patrick Wunderlich (@p-wunderlich)

Context​

ADR-0003 established that the physical-tenant (PT) id is stamped on the request before Spring Security's filter chain runs and is available throughout the request via PhysicalTenantContext.current() and the @PhysicalTenantId argument resolver. Non-prefixed cluster paths (/v2/...) fall back to default.

On top of that, a per-PT read infrastructure already exists on main:

  • ES/OS — PhysicalTenantSearchClientReadersConfiguration builds a Map<String, SearchClientReaders> (one reader bundle per configured PT, each bound to that PT's secondary-storage connection).
  • RDBMS — the equivalent per-PT Map<String, RdbmsTenantReaders> introduced in #54474.
  • ServiceRegistry exposes per-PT services (authorizationServices(physicalTenantId), userServices(physicalTenantId), …); CamundaSearchClients routes data reads to the in-context PT via withPhysicalTenant(id), which selects tenantReaders.get(id).

As a result the authorization query API is already PT-correct end to end: AuthorizationController.searchAuthorizations(@PhysicalTenantId …) → ServiceRegistry.authorizationServices(pt) → CamundaSearchClients (PT-scoped readers).

The gap is the authorization decision read — the lookup that the Camunda Security Library (CSL) performs to decide whether a principal may access a resource. That read is served by a singleton AuthorizationReader bean hardwired to the default PT:

  • ES/OS — SearchClientReaderConfiguration.authorizationReader() resolves the reader via requireDefault(...), i.e. the default entry of the per-PT reader map only.
  • RDBMS — RdbmsConfiguration builds the equivalent default-only AuthorizationReader.

Two call paths consume that singleton (#55252):

  1. Control-plane (Spring Security permission checks): hasPermission() → CSL ResourcePermissionService → AuthorizationRepositoryAdapter.findAuthorizations() → AuthorizationReader.search().
  2. Data-plane (result authorization inside CamundaSearchClients): doSearch/doGet → ResourceAccessDelegatingController → AuthorizationChecker → AuthorizationScopeRepositoryPort (SearchAuthorizationScopeRepository) → AuthorizationReader.

So even though a search for PT B's data reads from PT B's storage, the access check for that search consults PT default's authorization records. In a multi-PT deployment, authorization for every non-default PT is evaluated against the wrong tenant's data — on both paths, for both backends.

This decision is constrained by the Slice 1 (#54728) principle that CSL never learns about physical tenants: CSL exposes generic extension points (ports), and all PT resolution lives on the OC side in the host-supplied adapters and bean wiring. The basic-auth path already follows this shape — BasicAuthUserDetailsAdapter (an OC-supplied BasicAuthUserDetailsPort) resolves PhysicalTenantContext.current() and selects serviceRegistry.userServices(physicalTenantId) at call time. The authorization adapters are the missing counterpart.

A subtlety distinguishes the two paths. The control-plane check runs inside a Spring Security filter on the request thread, where PhysicalTenantContext.current() is guaranteed populated (per ADR-0003). The data-plane check runs inside CamundaSearchClients, whose ResourceAccessController chain is wired as a process-wide singleton shared across all per-PT CamundaSearchClients instances, and whose work may execute off the request thread (e.g. on the API-services executor) where the thread-bound PhysicalTenantContext does not propagate.

Decision​

Authorization reads resolve against the physical tenant in context, reusing the existing per-PT reader infrastructure — never a default-pinned singleton. All PT resolution stays on the OC side; CSL remains physical-tenant-agnostic. The resolution mechanism differs per path, matching where each check executes:

  1. Control-plane — lazy thread-context resolution in the OC-supplied CSL adapter. The adapter behind CSL's permission-check port resolves PhysicalTenantContext.current() and selects the in-context PT's authorization reader/service from the existing per-PT infrastructure (ServiceRegistry.authorizationServices(pt) / the per-PT reader map), exactly as BasicAuthUserDetailsAdapter does for users. The pre-security filter (ADR-0003) guarantees the context is set before the check runs.

  2. Data-plane — instance-bound resolution via the search client's PT. The authorization read on the ResourceAccessController path binds to the same PT the executing CamundaSearchClients instance is already scoped to (its currentPhysicalTenantId, captured on the request thread at withPhysicalTenant() time), not to a re-read of the thread-local. Concretely, the authorization repository/reader feeding the AuthorizationChecker is selected from the same per-PT SearchClientReaders the search client uses, so the access check and the data read always target the same PT. This removes the dependency on RequestContextHolder propagation and is robust to off-request-thread execution.

  3. Retire the default-pinned authorization singleton as the authZ-decision source on both backends (ES/OS SearchClientReaderConfiguration, RDBMS RdbmsConfiguration). A default-bound bean may remain only for genuinely cluster-scoped/legacy consumers; the authorization-decision path must go through PT-aware resolution.

  4. Apply uniformly to ES/OS and RDBMS, mirroring the per-PT reader resolution established in #54474.

The net contract: for any request carrying PT X (or default when unprefixed), both the Spring-Security permission check and the CamundaSearchClients result-authorization check read PT X's authorization records.

Consequences​

Positive​

  • Authorization is enforced against the correct PT's data on both paths and both backends, closing #55252.
  • Reuses the per-PT reader infrastructure already on main; no new storage-access machinery.
  • Keeps CSL physical-tenant-agnostic, consistent with Slice 1 — PT knowledge stays in OC adapters and wiring.
  • The control-plane adapter mirrors the accepted BasicAuthUserDetailsAdapter pattern, so the two identity read paths (users, authorizations) resolve PTs the same way.

Negative / trade-offs​

  • Two resolution mechanisms (thread-context for control-plane, instance-bound for data-plane) rather than one. This is deliberate — a single thread-local mechanism would be fragile on the data-plane's off-thread execution — but it is more to understand and document.
  • The data-plane change likely requires the ResourceAccessController / authorization repository to become per-PT (constructed alongside the per-PT readers) instead of a shared singleton, a larger wiring change than the control-plane adapter.
  • A default-tenant authorization bean that lingers for legacy consumers is a latent re-introduction of the bug if a new authZ-decision consumer binds to it by mistake; its remaining use sites must be enumerated and constrained.

Neutral​

  • No change to the authorization query API, which is already PT-routed.
  • Single/default-PT deployments are behaviourally unchanged (everything resolves to default).

Alternatives Considered​

  1. Thread-local resolution on both paths. Use PhysicalTenantContext.current() for the data-plane check too. Rejected: CamundaSearchClients work may run off the request thread, where the thread-bound context silently falls back to default — reintroducing the bug for exactly the multi-PT case this fixes.

  2. Carry the PT inside CSL's SecurityContext so a single mechanism serves both paths. Rejected: it makes CSL's context object physical-tenant-aware, violating the Slice 1 principle that CSL stays PT-agnostic, and couples the CSL contract to a PT concept it should not own.

  3. Per-request PT-aware proxy bean for AuthorizationReader. Expose the singleton as a request-scoped proxy that resolves the per-PT reader. Rejected for the data-plane for the same off-thread reason as (1); for the control-plane it is equivalent to but less explicit than resolving inside the adapter (the established pattern).

  4. Do nothing for now / defer past alpha3. Rejected as the standing decision: multi-PT authorization is incorrect without this, and the per-PT read infrastructure to fix it already exists. Final alpha3 scoping is confirmed on #55252 / #54728.
