# Dropdown — Carbon → shadcn migration guide

## Naming pitfall

The word "Dropdown" means different things in the two libraries:

- **Carbon `Dropdown`** = a single-select listbox with custom UI (closer to a `<select>`).
- **shadcn `DropdownMenu`** = an action menu (clickable items that fire callbacks, like a context menu — closer to Carbon's `OverflowMenu`).

The right migration target for Carbon's `Dropdown` is **shadcn's `Select`**, not `DropdownMenu`. This wrapper re-exports the canonical `Select` primitive (from `select/select.shadcn.tsx`) for that reason.

If you have Carbon `OverflowMenu` / `MenuButton` usages, those map to shadcn's `DropdownMenu` — covered separately under their own wrappers.

## Carbon Dropdown vs. Carbon Select

A quick reminder, because the user-facing UX is similar:
- **Carbon `Select`**: wraps a native `<select>`. Browser-styled picker on mobile, native form serialization.
- **Carbon `Dropdown`**: wraps a Downshift-based custom listbox. Custom UI everywhere, supports rich items via `itemToElement`, no native fallback.

shadcn collapses both into one primitive (`Select` = Radix custom listbox), so Carbon `Dropdown` and Carbon `Select` migrate to the same shadcn API. The distinguishing features (e.g., Carbon Dropdown's `itemToElement` for rich items) are picked up by shadcn `Select` because `<SelectItem>` accepts arbitrary children.

## Structural differences

Carbon's `Dropdown` is **monolithic with `items` + `itemToString` props**: pass a JS array, the component renders the listbox.

shadcn's `Select` is **fully decomposed**: `<Select>` → `<SelectTrigger>` + `<SelectValue>` → `<SelectContent>` containing `<SelectItem>`s.

| Concept | Carbon `Dropdown` | shadcn `Select` |
|---|---|---|
| Root | `<Dropdown items={…} itemToString={…} />` | `<Select>` (open + value state) |
| Trigger / value | implicit (built-in chevron + selected label) | `<SelectTrigger><SelectValue placeholder /></SelectTrigger>` |
| Items | `items` prop (array of objects) | `<SelectItem value={id}>{label}</SelectItem>` per option |
| Item label extractor | `itemToString(item)` | render the label as `<SelectItem>` children |
| Custom item rendering | `itemToElement(item)` | render rich content as `<SelectItem>` children |
| Field label | `titleText` | external `<label>` |
| Helper text | `helperText` | external `<p>` |
| Invalid + error text | `invalid` + `invalidText` | `aria-invalid` on `<SelectTrigger>` + external `<p>` |
| Warn state | `warn` + `warnText` | not modeled — render manually |
| Hide visible label | `hideLabel` | omit `<label>` and add `aria-label` to `<SelectTrigger>` |
| Direction | `direction="top" / "bottom"` | `<SelectContent side="top" / "bottom">` (passed through Radix) |
| Auto-align (collision) | `autoAlign` | always on via Radix Positioner |
| Type / appearance | `type="default" / "inline"` | not modeled — restyle `<SelectTrigger>` for inline |
| Density | `size="sm" / "md" / "lg"` | not modeled — restyle via Tailwind |
| Light variant | `light` boolean | not modeled — theme tokens |
| Initial selection | `initialSelectedItem` | `defaultValue` |
| Controlled | `selectedItem` + `onChange({selectedItem})` | `value` + `onValueChange(value: string)` |

## Carbon features missing in shadcn

- **`titleText` / `helperText` / `invalidText` / `warnText`** — Carbon ships labels and messaging on the field. shadcn forces external composition (typically with the `Field` primitive).
- **`warn` state** — Carbon distinguishes invalid (red) from warning (amber). shadcn has only `aria-invalid`. Mimic with hand-rolled amber classes.
- **`itemToString` / `itemToElement`** — Carbon takes complex item objects and a stringifier. shadcn `Select` works with `string` values; you map your objects yourself: `<SelectItem value={item.id}>{item.label}</SelectItem>`.
- **`type="inline"`** — Carbon's inline-label variant where the label sits beside the trigger. shadcn requires custom flexbox.
- **`size="sm" / "md" / "lg"`** — density variants. shadcn's `<SelectTrigger>` is one fixed size; restyle with Tailwind.
- **`light` prop** — Carbon's light-on-dark treatment. shadcn relies on theme tokens.
- **`renderSelectedItem`** — Carbon hook to re-render the trigger label. shadcn always uses the active `<SelectItem>`'s text via `<SelectItemText>` (text-only).
- **`downshiftProps`** — Carbon exposes the underlying Downshift instance. shadcn's Radix-based primitive doesn't expose internals; for custom keyboard or open behaviour you'd reach for the Radix props directly on `<Select>` / `<SelectTrigger>` / `<SelectContent>`.
- **Decorator slot (`decorator`)** — experimental Carbon prop for inline ai/icon decorators. No shadcn equivalent.
- **`<DropdownSkeleton>`** — Carbon ships a skeleton variant. shadcn has none.
- **Selection mode `selectionFeedback="top" / "fixed" / "top-after-reopen"`** — Carbon controls where the selected item appears after reopening. shadcn renders items in their array order always.

## shadcn features missing in Carbon

- **Decomposed structure** — render the trigger, content, items wherever you want. Carbon's slot order is fixed.
- **Group + separator primitives** — `<SelectGroup>`, `<SelectLabel>`, `<SelectSeparator>` for richer hierarchy. Carbon Dropdown supports a `<DropdownMenuItemGroup>` but it's awkward.
- **Per-item rich content** — render icons, descriptions, badges inside `<SelectItem>` directly without `itemToElement`.
- **`asChild`-style swap** — every sub-component accepts `asChild` (Radix slot pattern).
- **Animation primitives** — entry/exit animations driven by `data-state` attributes.
- **`position="popper" / "item-aligned"`** — choose between popper-style positioning (anchored under trigger) or item-aligned (selected item aligns with trigger). Carbon's listbox always pops below.
- **Scroll buttons** — auto-rendered `<SelectScrollUpButton>` / `<SelectScrollDownButton>` on overflow. Carbon's listbox uses native scroll only.

## Behavioural differences

- **Underlying primitive** — Carbon: Downshift-based custom listbox. shadcn: Radix Select (`@radix-ui/react-select`). Both are custom UIs (not native `<select>`).
- **Form integration** — neither serialises natively; both require either a hidden `<input>` mirror or form-library integration.
- **Keyboard** — Both support arrow keys, type-ahead, Escape. shadcn additionally exposes Radix's lifecycle callbacks.
- **Z-index / portal** — shadcn portals to `<body>` by default; Carbon renders inline.

## Migration checklist

1. Replace `<Dropdown items={items} itemToString={…} titleText helperText />` with the shadcn composition (see `Default` story for the canonical shape).
2. Move `titleText` to a sibling `<label>`, `helperText` to a sibling `<p>`, `invalidText` to a sibling `<p>` toggled by `aria-invalid`.
3. Replace `items` + `itemToString` + `itemToElement` with explicit `{items.map((item) => <SelectItem value={item.id}>{render(item)}</SelectItem>)}`.
4. `selectedItem` → `value`; `initialSelectedItem` → `defaultValue`. Note: Select uses a `string` value; map back to your object via your `items` array.
5. `onChange({selectedItem})` → `onValueChange(value: string)`.
6. For `invalid`: add `aria-invalid` + `className="border-destructive focus:ring-destructive"` to `<SelectTrigger>`.
7. For `warn`: hand-roll amber classes — no shadcn primitive.
8. For `direction="top"`: pass `side="top"` to `<SelectContent>`.
9. If you used `itemToElement` for rich items: just put rich content as children of `<SelectItem>`. The selected-value display is text-only via `<SelectItemText>`; if you need rich content in the trigger too, override `<SelectValue>` with a custom render.
10. Drop `size`, `type`, `light` — restyle with Tailwind classes if needed.
11. If form serialisation is needed: add `<input type="hidden" name>` synced with the selected value, or wire up `react-hook-form`'s `Controller`.
12. **Don't migrate Carbon `Dropdown` to shadcn `DropdownMenu`** — that's the action-menu primitive, mapped to Carbon's `MenuButton` / `OverflowMenu`.
