# Checkbox — Carbon → shadcn migration guide

## Structural differences

Carbon's `Checkbox` is a **monolithic field** — input, label, helper text, error text, and the visual box all live inside one component, configured by props (`labelText`, `helperText`, `invalidText`, `warnText`).

shadcn's `Checkbox` is **just the box** — a Radix primitive with no built-in label or messaging. You compose label + error text yourself (typically with the shadcn `Label` and `Field` primitives, or plain `<label>`).

| Concept | Carbon | shadcn |
|---|---|---|
| Wrapper / box | `<Checkbox labelText="…" />` | `<Checkbox />` (no label) |
| Label | `labelText` prop | wrap in `<label>` or use `<Label htmlFor>` |
| Helper text | `helperText` prop | render an extra `<p>` / use `<FieldDescription>` |
| Error text | `invalidText` + `invalid` | render an extra `<p>` + add `aria-invalid` and red `border-destructive` classes |
| Warning text | `warnText` + `warn` | not modeled — render manually |
| Indeterminate | `indeterminate` (uncontrolled-friendly) | controlled only: `checked={'indeterminate'}` + `onCheckedChange` |
| Read-only | `readOnly` | not modeled — use `disabled` or set `tabIndex={-1}` + visual class |
| Hide visible label | `hideLabel` | render no `<label>` text (still set `aria-label` for SR users) |

## Carbon features missing in shadcn

- **`labelText`** — Carbon's checkbox carries its own label. shadcn forces external composition.
- **`helperText` / `invalidText` / `warnText`** — built-in messaging slots in Carbon. shadcn has none; layouts handled by `Field` primitive (separate package) or by hand.
- **`warn` state** — Carbon distinguishes `invalid` (error) from `warn` (yellow). shadcn has no warning concept; mimic with `aria-invalid="true"` + custom amber classes.
- **`readOnly`** — Carbon greys the box but keeps it focusable and announced. shadcn has no equivalent — `disabled` removes it from the tab order.
- **`indeterminate` as a prop** — Carbon accepts it on uncontrolled checkboxes. shadcn requires you to manage `checked` state and pass the literal string `'indeterminate'`.
- **`title` / `wrapperClassName`** — Carbon offers wrapper-level class hooks for the form-field shell. shadcn has no shell to target.
- **`CheckboxSkeleton`** — Carbon ships a skeleton variant. shadcn has none — use the generic `skeleton` primitive.

## shadcn features missing in Carbon

- **Headless / unstyled API** — shadcn exposes Radix's `data-state="checked|unchecked|indeterminate"` attributes, so you can re-skin entirely with Tailwind. Carbon's CSS is fixed.
- **Composability with `Field` / `Label`** — pair with shadcn's `Field`, `FieldGroup`, `FieldDescription` to build complex forms; the box is decoupled.
- **Form integration** — works out-of-the-box with React Hook Form, TanStack Form, etc. Carbon's `labelText`-based API is awkward to wire into form libraries.

## Behavioural differences

- **Native vs. faux input** — Carbon renders a native `<input type="checkbox">` and styles it with sibling pseudo-elements. shadcn renders a Radix `<button role="checkbox">` (no native input). For form serialization you may need a hidden `<input>` companion or an integration like `react-hook-form`.
- **Focus ring** — Carbon: 2px inset blue. shadcn: 2px outset ring with offset.
- **Border radius** — Carbon: `0` (square). shadcn: `rounded-sm` (2px).
- **Icon** — Carbon: CSS-drawn checkmark. shadcn: `Check` icon from `lucide-react`.
- **Click target size** — Carbon: 24×24 with 8px hit padding via label. shadcn: 16×16 box; expand the click target via the surrounding label or by sizing classes.

## Migration checklist

1. Replace `<Checkbox labelText="…" />` with `<label className="flex items-center gap-2"><Checkbox /><span>…</span></label>` (or `<Label htmlFor>`).
2. Move `helperText`, `invalidText`, `warnText` into sibling `<p>` / `<span>` elements.
3. For `invalid`: add `aria-invalid` + `className="border-destructive data-[state=checked]:bg-destructive"`.
4. For `warn`: render your own amber styling — no shadcn primitive exists.
5. For `indeterminate`: switch to controlled `checked` + `onCheckedChange` with `'indeterminate'` value.
6. Replace `readOnly` with `disabled` (or build a custom non-focusable visual). Loses readonly semantics.
7. If the form relies on native `<input type="checkbox">` serialization, add a hidden input or use `react-hook-form`.
