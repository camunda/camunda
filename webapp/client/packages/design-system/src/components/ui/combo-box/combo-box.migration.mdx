# ComboBox — Carbon → shadcn migration guide

## Library origin

shadcn's combobox in this project is the **new-york-v4** registry variant — built on **Base UI** (`@base-ui/react`), **not** Radix UI. It composes with the shadcn `InputGroup` primitive (a styled wrapper around a flex row with addon slots) and Base UI's headless `Combobox.*` parts. The "popover + command" composition shown elsewhere in shadcn docs is the legacy approach; the registry now ships an actual `Combobox` component.

## Structural differences

Carbon's `ComboBox` is a **monolithic field**: input, list, label, helper text, error text, and selection state all live inside one component, configured by props (`items`, `itemToString`, `titleText`, `helperText`, `invalidText`).

shadcn's `Combobox` is **fully decomposed** into 15+ sub-components (`Combobox`, `ComboboxValue`, `ComboboxInput`, `ComboboxContent`, `ComboboxList`, `ComboboxItem`, `ComboboxGroup`, `ComboboxLabel`, `ComboboxEmpty`, `ComboboxSeparator`, `ComboboxChips`, `ComboboxChip`, `ComboboxChipsInput`, `ComboboxTrigger`, `ComboboxClear`). You compose the structure yourself.

| Concept | Carbon | shadcn |
|---|---|---|
| Root | `<ComboBox items={…} />` | `<Combobox items={…}>…</Combobox>` |
| Trigger / input | implicit (built-in) | `<ComboboxInput>` (text input + dropdown trigger via InputGroup) |
| Dropdown panel | implicit | `<ComboboxContent>` (Base UI Portal + Positioner + Popup) |
| Items collection | `items` prop + `itemToString` | `<ComboboxList>{(item) => <ComboboxItem>}` (render-prop) |
| Empty state | `noItemsFoundText` (custom prop) | `<ComboboxEmpty>` |
| Field label | `titleText` | external `<label>` |
| Helper text | `helperText` | external `<p>` |
| Error text / invalid | `invalid` + `invalidText` | `aria-invalid` + external `<p>`; `InputGroup` reads `aria-invalid` and styles itself |
| Warning state | `warn` + `warnText` | not modeled — render manually |
| Filter / search | `shouldFilterItem` callback | Base UI filters automatically by typed text; override via `filter` prop on `<Combobox>` |
| Multi-select chips | `multiselect` flag (separate component in Carbon: `MultiSelect`) | `<ComboboxChips>` + `<ComboboxChip>` + `<ComboboxChipsInput>` |
| Clear selection | `selectedItem={null}` from outside | `<ComboboxClear>` (renders an X button inside InputGroup) |

## Carbon features missing in shadcn

- **`titleText` / `helperText` / `invalidText` / `warnText`** — Carbon's combobox carries its own labels and messaging slots. shadcn forces external composition, typically via the `Field` primitive.
- **`warn` state** — Carbon distinguishes invalid (red) from warning (yellow/amber). shadcn has only `aria-invalid` (destructive); mimic warning by hand-rolling amber border classes.
- **`shouldFilterItem(input, item)`** — Carbon exposes a per-item filter callback. Base UI filters by the rendered text of `ComboboxItem` automatically; for custom filtering pass a `filter` prop on `<Combobox>` that takes `(item, query) => boolean`.
- **`itemToElement`** — Carbon supports a `(item) => ReactNode` to render rich items (icon + label). shadcn's `ComboboxItem` accepts arbitrary children, but the default `Indicator` (the check icon) sits absolute-positioned on the right; complex layouts may need re-styling.
- **`type="default" | "inline"`** — Carbon's inline combobox sits flush with surrounding text. shadcn has no inline variant; build it with raw Base UI parts and minimal padding/border.
- **`size="sm" | "md" | "lg"`** — Carbon offers density variants. shadcn relies on Tailwind classes on `<ComboboxInput>` / `<InputGroup>` for sizing.
- **`light` prop** — Carbon's "light" variant on dark backgrounds. shadcn's theming is already token-based — no equivalent prop needed; set CSS variables instead.
- **`ComboBoxSkeleton`** — Carbon ships a skeleton state. shadcn has none — use the generic `skeleton` primitive.
- **Built-in `selectionFeedback="top" | "fixed" | "top-after-reopen"`** — controls where the selected item appears after reopening. shadcn has no equivalent; selection ordering is determined by your `items` array.
- **Allow custom value (`allowCustomValue`)** — Carbon lets you accept text not in the list. Base UI exposes this through `<Combobox openOnInputClick={…}>` and treating typed input as a value, but the wiring is up to you.

## shadcn features missing in Carbon

- **`Combobox.Chips` + `Combobox.Chip`** — first-class multi-select with inline chips and remove handles. In Carbon you'd use the separate `MultiSelect` component for this.
- **`anchor` prop on `<ComboboxContent>`** — anchor the dropdown to a different element (e.g., a global "filter" pill). Carbon has no headless positioning.
- **`render` prop / asChild composition** — every Base UI part accepts `render={…}` so you can swap the underlying element entirely while keeping behaviour. Powers things like `<ComboboxClear>` reusing `<InputGroupButton>`.
- **`useComboboxAnchor()` hook** — utility to anchor the dropdown to an arbitrary ref (useful for chip-style layouts).
- **Headless filtering / custom item shapes** — `Combobox` works with any item shape; you tell it how to extract the label via `itemToStringLabel`. Carbon expects either a primitive list or a list of `{label, …}` objects with `itemToString`.

## Behavioural differences

- **Underlying primitive** — Carbon: native `<input>` + a popper. shadcn: Base UI's `<input>` rendered through `InputGroup` + a Portal'd Popup positioned by `ComboboxPrimitive.Positioner`. Z-index, scroll-locking, and focus-trap behaviour come from Base UI, not Radix.
- **Default open trigger** — Carbon: typing or clicking the chevron. shadcn: same, but typing is wired through Base UI's controller; the "chevron" is `<ComboboxTrigger>` that toggles open state.
- **Selection** — Carbon: `selectedItem` (single object) + `onChange({selectedItem})`. shadcn: `value` / `onValueChange` (single or array depending on `multiple`).
- **Form integration** — Carbon's `<input>` is part of the DOM and serializes natively. shadcn's `Combobox` is also a real `<input>` (good), but the *selected item* is JS-only; for form serialization, use a hidden `<input>` mirror or integrate with `react-hook-form`.
- **Focus ring** — Carbon: 2px inset blue. shadcn: 1px ring colour from `--ring` via `InputGroup` `has-[…]:ring-ring`.

## Migration checklist

1. Replace `<ComboBox titleText helperText invalidText items itemToString placeholder selectedItem onChange />` with the shadcn composition (see `Default` story for the canonical shape).
2. Move `titleText` to a sibling `<label>`. Move `helperText` and `invalidText` to sibling `<p>`s; toggle `aria-invalid` on `<ComboboxInput>` for the error state.
3. Replace `itemToString` with `itemToStringLabel` on `<Combobox>` and a render-prop `<ComboboxList>{(item) => <ComboboxItem value={item}>…</ComboboxItem>}`.
4. Replace `noItemsFoundText` with `<ComboboxEmpty>…</ComboboxEmpty>` inside `<ComboboxContent>`.
5. Replace `shouldFilterItem` with a `filter={(item, query) => …}` prop on `<Combobox>`.
6. For multi-select Carbon `MultiSelect`: switch to `<ComboboxChips>` + `<ComboboxChipsInput>` + `<ComboboxChip>` (single component family, not a separate one).
7. For `warn` state: add amber Tailwind classes to `<InputGroup>` manually — no shadcn primitive for warning.
8. If the form library reads `selectedItem` as an object, mirror it in a hidden `<input>` or move to `react-hook-form` with a `Controller`.
9. The `light` prop is unnecessary — let the surrounding theme tokens handle it.
