# Modal — Carbon → shadcn migration guide

## Two Carbon APIs, one shadcn API

Carbon ships **two** parallel modal APIs:
- **`Modal`** — monolithic, prop-driven (`modalHeading`, `modalLabel`, `primaryButtonText`, `secondaryButtonText`, `onRequestSubmit`, `onRequestClose`).
- **`ComposedModal`** + `ModalHeader` + `ModalBody` + `ModalFooter` — explicit composition where each piece is a separate component.

shadcn ships **one** API: `Dialog` family, always composed:
```
<Dialog>
  <DialogTrigger asChild>{trigger}</DialogTrigger>
  <DialogContent>
    <DialogHeader>
      <DialogTitle>…</DialogTitle>
      <DialogDescription>…</DialogDescription>
    </DialogHeader>
    {body}
    <DialogFooter>
      <DialogClose asChild><Button variant="outline">Cancel</Button></DialogClose>
      <Button>Save</Button>
    </DialogFooter>
  </DialogContent>
</Dialog>
```

Both Carbon `Modal` and `ComposedModal` migrate to the same shadcn `Dialog` shape. The `ComposedModal` migration is more direct (1:1 component mapping); the `Modal` migration replaces props with explicit children.

## Decision rule: Dialog vs. AlertDialog

Carbon's `<Modal danger>` (or destructive `ComposedModal`) is for "are you sure?" confirmations. In shadcn, **the right primitive for those is `AlertDialog`**, not `Dialog`. The differences:

| | shadcn `Dialog` | shadcn `AlertDialog` |
|---|---|---|
| Use case | general modal (forms, info, multi-step) | destructive / blocking confirmation |
| Built-in cancel button | no — render `<DialogClose>` manually | yes — `<AlertDialogCancel>` auto-closes |
| Built-in action button | no — render any `<Button>` | yes — `<AlertDialogAction>` auto-closes after handler |
| Default focus | first focusable element | the cancel button (safer default for destructive) |
| Dismissible by clicking outside | yes | no (forces explicit choice) |
| Dismissible by Escape | yes | yes |

**Migration rule:** if the Carbon modal is destructive (`danger` prop) or it's a binary "confirm/cancel" prompt, switch to `AlertDialog` (covered in `actionable-notification/`). Otherwise use `Dialog`.

## Mapping

### `Modal` (monolithic) → `Dialog`

| Carbon `Modal` prop | shadcn |
|---|---|
| `open` | `open` on `<Dialog>` |
| `onRequestClose` | `onOpenChange={(o) => !o && handleClose()}` on `<Dialog>` |
| `modalHeading` | `<DialogTitle>` content |
| `modalLabel` | small `<p>` above the title (no first-class slot) |
| children (body) | content between `<DialogHeader>` and `<DialogFooter>` |
| `primaryButtonText` + `onRequestSubmit` | `<Button onClick={handleSubmit}>{label}</Button>` in `<DialogFooter>` |
| `secondaryButtonText` | `<DialogClose asChild><Button variant="outline">{label}</Button></DialogClose>` |
| `danger` | switch to `AlertDialog` (or use `<Button variant="destructive">` for the action) |
| `size="xs" / "sm" / "md" / "lg"` | restyle `<DialogContent className="sm:max-w-md / lg / 2xl / 4xl">` |
| `passiveModal` (no buttons) | omit `<DialogFooter>` |
| `preventCloseOnClickOutside` | `<DialogContent onPointerDownOutside={(e) => e.preventDefault()}>` |
| `loadingStatus="active"` | render `<Spinner />` next to the action button manually |
| `selectorPrimaryFocus` | `<DialogContent onOpenAutoFocus={…}>` then `.focus()` your target |

### `ComposedModal` → `Dialog` (1:1 component mapping)

| Carbon | shadcn |
|---|---|
| `<ComposedModal open onClose>` | `<Dialog open onOpenChange={(o) => !o && onClose()}>` |
| `<ModalHeader title label closeModal>` | `<DialogHeader><DialogTitle>{title}</DialogTitle></DialogHeader>` (label as a sibling `<p>`; close handled by `<DialogContent>`'s built-in X) |
| `<ModalBody>` | content directly inside `<DialogContent>` (or wrap in a `<div>` for layout) |
| `<ModalFooter primaryButtonText secondaryButtonText>` | `<DialogFooter>` with explicit `<Button>`s |

## Carbon features missing in shadcn

- **`size="xs" / "sm" / "md" / "lg"`** — Carbon's modal width variants. shadcn `<DialogContent>` defaults to `sm:max-w-lg`; restyle with `sm:max-w-md` (sm), `sm:max-w-2xl` (lg), `sm:max-w-4xl` (xl) etc.
- **`passiveModal`** — Carbon prop to render the modal without footer buttons. shadcn: just don't render `<DialogFooter>`.
- **`hasScrollingContent`** — Carbon flag for when the body scrolls. shadcn: scrolling is automatic when content overflows; tweak with `<div className="max-h-[60vh] overflow-y-auto">` inside `<DialogContent>`.
- **`loadingStatus` / `loadingDescription`** — Carbon's in-modal loading state. shadcn: render `<Spinner />` next to the action button; manage state yourself.
- **`shouldSubmitOnEnter`** — Carbon submits the form on Enter. shadcn: wrap content in `<form onSubmit>`; Enter inside any form input submits.
- **`selectorPrimaryFocus`** / `selectorsFloatingMenus`** — Carbon's CSS-selector-based focus management. shadcn: use `onOpenAutoFocus` callback + `ref.current.focus()`.
- **`alert`** — Carbon's prop to use `role="alertdialog"`. shadcn: use `<AlertDialog>` instead.
- **Built-in danger styling** — Carbon `danger` prop colours the primary button red and adds an alert icon. shadcn: use `<AlertDialog>` (recommended) or set `<Button variant="destructive">` manually.
- **`preventCloseOnClickOutside`** — `<DialogContent onPointerDownOutside={(e) => e.preventDefault()}>` (Radix prop).
- **Skeleton variant** — Carbon ships `<ModalSkeleton>`. shadcn has none — use the generic `skeleton` primitive inside the dialog.
- **`launcherButtonRef`** — Carbon's prop for explicit focus return on close. shadcn: Radix returns focus to the trigger automatically; for custom return use `onCloseAutoFocus={(e) => { e.preventDefault(); ref.current.focus() }}`.

## shadcn features missing in Carbon

- **`asChild` on the trigger** — `<DialogTrigger asChild>{anyElement}</DialogTrigger>` lets you keep your trigger as any element (button, link, custom component).
- **Always-portalled content** — shadcn's `<DialogContent>` renders into `<body>` so it's never clipped by ancestor `overflow:hidden` or `transform`. Carbon's modal also portals (in modern versions), but Radix is more aggressive.
- **`onPointerDownOutside` / `onEscapeKeyDown` / `onInteractOutside`** — fine-grained dismissal control. Carbon collapses these into `onRequestClose`.
- **`onOpenAutoFocus` / `onCloseAutoFocus`** — lifecycle hooks for focus management.
- **Built-in close button on `<DialogContent>`** — auto-rendered top-right X with proper `<span className="sr-only">Close</span>`. Carbon's monolithic `Modal` has one too; `ComposedModal` requires you to wire it via `<ModalHeader closeModal>`.
- **Tailwind composability** — every part accepts `className`; restyle inline.
- **Dialog as content for routing** — easy to drive open state from URL params or React Router `<Outlet>` / `<Modal>` patterns. Carbon supports this too but needs more wiring.

## Behavioural differences

- **Default width** — Carbon: ~480px (md). shadcn: `sm:max-w-lg` (~512px), full-width on mobile with 1rem margin.
- **Default height** — Carbon: viewport-bounded. shadcn: same; long content scrolls.
- **Animation** — Both: scale + fade in. shadcn animations are driven by `data-state` attributes via Tailwind animate utilities.
- **Focus trap** — Both trap focus inside the open dialog.
- **Body scroll lock** — Both lock body scroll when open.
- **Z-index** — Both render at high z-index. shadcn: `z-50` on overlay + content.

## Migration checklist

1. **Decide:** is it destructive / a "confirm" prompt? If yes, migrate to `AlertDialog` (`actionable-notification/` wrapper) instead of `Dialog`.
2. Replace `<Modal open onRequestClose>` with `<Dialog open onOpenChange={(o) => !o && onRequestClose()}>`.
3. For uncontrolled use: `<DialogTrigger asChild>{trigger}</DialogTrigger>` instead of an `open` boolean.
4. Move `modalHeading` to `<DialogTitle>`; `modalLabel` to a small `<p>` above the title.
5. Wrap title/label in `<DialogHeader>`, body content directly inside `<DialogContent>`, action buttons in `<DialogFooter>`.
6. Replace `primaryButtonText` + `onRequestSubmit` with `<Button onClick={…}>{text}</Button>`.
7. Replace `secondaryButtonText` with `<DialogClose asChild><Button variant="outline">{text}</Button></DialogClose>` — `DialogClose` auto-closes the dialog.
8. For `passiveModal`: omit `<DialogFooter>`.
9. For `size`: restyle via `<DialogContent className="sm:max-w-md/2xl/4xl">`.
10. For `preventCloseOnClickOutside`: `<DialogContent onPointerDownOutside={(e) => e.preventDefault()}>`.
11. For `selectorPrimaryFocus`: `<DialogContent onOpenAutoFocus={(e) => { e.preventDefault(); ref.current?.focus() }}>`.
12. For `ComposedModal` family: 1:1 mapping (`ModalHeader` → `DialogHeader`+`DialogTitle`, `ModalBody` → bare children, `ModalFooter` → `DialogFooter` with explicit buttons).
13. If you used `loadingStatus`: render a `<Spinner />` next to the action button and disable it while pending.
14. **For destructive (`danger`) modals: switch to `AlertDialog`** — it has the right semantics (`role="alertdialog"`, no outside-click dismiss, defaults focus to cancel).
