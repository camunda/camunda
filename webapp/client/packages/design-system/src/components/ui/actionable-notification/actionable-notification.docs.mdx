# ActionableNotification

A status banner with an inline call-to-action. Use when the user must read a status message and can act on it directly from the message itself, without leaving their flow.

## When to use

- **Inline status with a CTA** — "Unsaved changes — Save now", "New version — Reload". Compose `Alert` + `AlertTitle` + `AlertDescription` + a `Button`.
- **Recoverable error with retry** — operation failed, user can re-run it. Use `<Alert variant="destructive">` with a `<Button variant="destructive">` action.
- **Blocking confirmation** — destructive or irreversible action that needs explicit confirmation. Use `AlertDialog` with `AlertDialogAction` and `AlertDialogCancel`.
- **Dismissible advisory** — non-critical info the user can close. Compose `Alert` with your own `useState` open flag and an `<Button variant="ghost" size="icon">` X.

## When not to use

- **Transient, non-blocking feedback** ("Saved", "Copied") — use `Sonner` toasts.
- **Static informational text with no action** — use plain `Alert` without a button.
- **Form validation errors tied to a single field** — use the field's inline error message, not a banner.

## Composition

- The shadcn equivalent is **decomposed**: there is no monolithic `ActionableNotification`. Assemble `Alert` + icon + `Button` in a flex row yourself.
- Status colours beyond `default` and `destructive` (success, warning, info) require extending `alertVariants` or applying Tailwind classes — they are not built in.
- Dismiss state is owned by you. Wrap the `Alert` in a `useState` flag and conditionally render.
